Writing an analysis

From a question to a result table

An “analysis” here is a small MATLAB function that takes some datasets, computes numbers, and hands back a table — plus a bit of wiring so the pipeline can run it for you on every matching dataset and store the results. Four files, one pattern. We’ll follow a real one: SMS_CA (spots-multi-size, cell-attached).

The idea1 · The function2 · The template3 · The result table4 · The runnerRunning itChecklist

The shape of it

Three pieces do the work, and one ties them together:

the maths

An analysis function

Plain MATLAB. In: datasets. Out: a table with one row per dataset.

the shape

A result-table template

A three-column spreadsheet naming each output field and its type.

the store

A result table

A DataJoint table (sln_results.Dataset…) that holds the rows.

the trigger

A runner

A computed table that finds matching datasets and calls your function on each.

1 · Write the analysis function

It receives a data group (a set of datasets), asks the template for an empty table of the right shape, then fills one row per dataset. Keep the identifying fields (file_name, dataset_name, source_id) — the pipeline needs them to know which cell each row belongs to.

DJ_analysis_Greg/SMS_CA.m (trimmed)
function R = SMS_CA(data_group, params)
% One row of results per dataset in data_group.
if nargin < 2, binSize = 10; else, binSize = params.binSize; end

% 1. Which datasets are we working on?
datasets = aka.Dataset & data_group;           % restrict to the datasets handed in
datasets_struct = fetch(datasets);
N_datasets = datasets.count;

% 2. Ask for an empty results table shaped like the template.
R = sln_results.table_definition_from_template('SMSCA', N_datasets);

% 3. Fill it in, one dataset (one row) at a time.
for d = 1:N_datasets
    % ... pull this dataset's epochs, count spikes per spot size ...
    R.file_name{d}       = datasets_struct(d).file_name;
    R.dataset_name(d)    = datasets_struct(d).dataset_name;
    R.source_id(d)       = datasets_struct(d).source_id;
    R.spot_sizes{d}      = spot_sizes';
    R.spikes_stim_mean{d}= spikes_stim_mean;
    R.baseline_rate_hz(d)= baseline_rate;
end
end
The one helper to remember sln_results.table_definition_from_template('SMSCA', N) reads result_table_templates/SMSCA.xlsx and hands you an empty N-row MATLAB table with the right columns and types. You just fill it.

2 · Make the template

The template is the single source of truth for your output columns. You can type the spreadsheet by hand, but the easy way is to build one example table R in MATLAB (with variable descriptions set), then let the lab helper write the file for you:

in MATLAB
sln_results.write_table_template(R, 'SMSCA');
% writes result_table_templates/SMSCA.xlsx from R's columns
What a template looks like A template is a tiny spreadsheet with three columns. Each row describes one output field:
FieldTypeDescription
dataset_namestringdataset name
source_iduint16identifies the cell
baseline_rate_hzdoublebaseline firing rate (Hz)
sms_psthcellthe PSTH image

Type is a MATLAB variable type: string and double for single values, cell for anything with a vector or matrix in each row (it becomes a longblob in the table).

3 · Define the result table

This is the DataJoint table that actually stores results, in the sln_results schema. Its header lists the same identifying fields above the --- line and your output fields below it, plus three housekeeping fields the pipeline fills automatically: user, entry_time, and git_tag (so every result records who made it, when, and with which version of the code).

schemas/+sln_results/DatasetSMSCA.m (trimmed)
%{
# DatasetSMSCA
file_name    : varchar(128)   % file name from symphony
dataset_name : varchar(128)   % dataset name
source_id    : int unsigned   % identifies the cell the dataset belongs to
---
-> sln_lab.User                                % who computed this
baseline_rate_hz=NULL : float                  % baseline firing rate (Hz)
entry_time = CURRENT_TIMESTAMP : timestamp     % when it was computed
git_tag : varchar(128)                         % code version used
spot_sizes=NULL   : longblob                   % set of spot sizes (microns)
sms_psth=NULL     : longblob                   % the full PSTH image
%}
classdef DatasetSMSCA < dj.Manual
end
Keep them in sync The template columns and the table’s below-the-line fields describe the same outputs. Change one, change the other. sln_results.createTableFromTemplate can scaffold the table file from a template so they match to begin with.

4 · Write the runner

The runner is a computed table — it’s what makes the whole thing automatic. Its keySource is a query describing which datasets this analysis applies to (here: cell-attached “spots multi size” recordings). For every matching dataset that hasn’t been done yet, DataJoint calls makeTuples once, where you run your function and store the result.

schemas/+sln_results/SMSCARunner.m (trimmed)
%{
# SMSCARunner
-> sln_symphony.Dataset
---
analysis_name : varchar(128)
analysis_entry_time = CURRENT_TIMESTAMP : timestamp
git_tag : varchar(128)
%}
classdef SMSCARunner < dj.Computed
    properties
        % keySource = the datasets this analysis SHOULD run on:
        % cell-attached "spots multi size" recordings.
        keySource = sln_symphony.Dataset & ( ...
            sln_symphony.Dataset * sln_symphony.DatasetEpoch * aka.Epoch * ...
            sln_symphony.ExperimentEpochBlock * sln_symphony.ExperimentElectrode & ...
            'protocol_name="spots_multi_size"' & 'amp_mode="Cell attached"')
    end
    methods(Access=protected)
        function makeTuples(self, key)      % called once per un-computed dataset
            key.analysis_name = 'SMS_CA';
            R = SMS_CA(key);                                 % run the analysis
            sln_results.insert(R, 'Dataset', 'false');       % write the results table
            q = sln_results.DatasetSMSCA & key & 'LIMIT 1 PER source_id ORDER BY entry_time DESC';
            key.git_tag = fetch1(q, 'git_tag');
            self.insert(key);                                % mark this dataset done
        end
    end
end
Two helpers here sln_results.insert(R, 'Dataset', 'false') writes your filled table R into the matching sln_results.Dataset… table. self.insert(key) then records that this dataset is finished, so it won’t be recomputed.

Running it

Once the four files exist, you don’t call anything by hand for each cell. You populate the runner and DataJoint marches through every un-computed dataset:

in MATLAB
populate(sln_results.SMSCARunner);   % just yours
% or, on the server, everything that's out of date:
populate_computed_tables;

Then read your results back like any other table — restricting by cell, animal, or genotype as needed:

in MATLAB
R = sln_results.DatasetSMSCA & 'source_id = 3';
rates = fetch(R, 'baseline_rate_hz', 'sms_psth');

Your checklist for a new analysis

Name it

Pick a short name like MovingBar_CA. The _CA/_CC/_VC suffix says the recording mode (cell-attached, current-clamp, voltage-clamp).

Write the function

DJ_analysis_Greg/<Name>.m — in: a data group; out: a table from the template with one row per dataset. Steal the closest existing analysis as a starting point.

Create the template

Build an example R, then sln_results.write_table_template(R, '<Name>').

Create the result table

schemas/+sln_results/Dataset<Name>.m, identifying fields on top, outputs plus user/entry_time/git_tag below.

Write the runner

<Name>Runner.m as a dj.Computed table; set keySource to the datasets it applies to.

Populate & check

populate(sln_results.<Name>Runner), then fetch a few rows to sanity-check before running it lab-wide.

Best first move Copy the analysis whose data looks most like yours (there are ~40 in DJ_analysis_Greg/ with matching templates in result_table_templates/) and change it a piece at a time. Reading two or three of them side by side teaches the pattern faster than any single explanation.
