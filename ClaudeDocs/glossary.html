Glossary

The words you’ll keep hearing

Plain definitions for the terms that come up around this database. None of them require knowing SQL — that’s the point of DataJoint.

DataJointthe tool
The framework that runs this whole database. You describe data in MATLAB and DataJoint turns it into a real MySQL database — and writes all the SQL for you. You almost never write SQL yourself.
Schemaa folder of tables
A named group of related tables, like sln_symphony (recordings) or sln_animal (the colony). In the code, one schema is one +folder.
Tablea grid of rows
A grid where every row is one thing (one epoch, one cell) and every column is one fact about it. In MATLAB you refer to a table as schema.TableName.
Primary keywhat makes a row unique
The column(s) that identify a row, shown above the dashed line in a table’s definition and marked with a dot on the table cards. No two rows can share the same primary key.
Foreign keya link to a parent
A field that says “this row belongs to that row in another table”, written -> schema.Table. Foreign keys are the arrows in the levels ladder — they carry the parent’s key down into the child.
Restrictkeep only some rows
Narrowing a table to the rows you want, with the & operator: sln_symphony.Dataset & 'source_id = 3'. Like a filter.
Joincombine two tables
Gluing two tables together on their shared key, with the * operator. Because tables share keys, DataJoint knows how to line them up — you never spell out the join by hand.
Projectionpick / rename columns
Choosing or renaming columns with proj(...). Handy when two tables both call something source_id and you need to tell them apart.
Fetchpull data into MATLAB
Actually reading rows out of the database into MATLAB variables: fetch(query, 'baseline_rate_hz'). A query on its own doesn’t touch the database until you fetch (or count).
Populatecompute the missing rows
Telling a computed table to run its analysis on every input that doesn’t have a result yet: populate(sln_results.SMSCARunner). The engine behind Writing an analysis.
Tierwhat kind of table
How a table gets its data. Manual: typed in by people or apps. Lookup: a fixed reference list. Imported: read from raw files. Computed: produced by an analysis. Part: a piece that only exists with its parent. Each has a color on the table cards.
Dataseta curated group of epochs
A hand-picked set of trials for one analysis — the unit almost every result is built on. Not to be confused with the whole recording file. Made in the DataGrouper app.
Epochone trial
A single presentation of a stimulus, with a start time and a duration. The smallest unit of the recording. Many epochs make a dataset.
Protocola stimulus program
The stimulus that was run — spots_multi_size, moving_bar, and so on. Its parameters live in the auto-generated …bp / …ep tables you’ll see grouped on the Tables page.
aka.*friendly shortcuts
The +aka folder holds aliases that bundle common queries under a plain name — aka.Dataset, aka.Epoch, aka.SMSparams. Analyses use these so you don’t rebuild the same join every time.
git tagthe code version
A stamp of which version of the code produced a result, stored on every result row. It’s how the lab can tell whether a number needs recomputing after a bug fix.
Still stuck on a word? The official DataJoint documentation covers the query operators (&, *, proj) in more depth, with examples in both MATLAB and Python.
